# Medical Fine-Tune Plan — User-Led, Post-Corpus

**Date:** 2026-04-27 · **Status:** research-only, **user-led** ·
**Owner:** Brandon Dent, MD (corpus + decision); assistant only scopes.

This brief documents the path the user chose over BioNeMo or MedGemma:
**fine-tune Nemotron-Nano-30B-A3B on a user-built medical corpus**.
The action of training is the user's; this doc just captures the
shape so a future session has the context.

## 1. Why this path

The user's question on the table was "how do we raise medical
accuracy on patient-encounter reasoning." The three options were:

1. **BioNeMo** — biomolecular, not clinical-encounter. Category error.
   Drop. (See [`bionemo-deferred.md`](#) — never written; we agreed
   not to add it.)
2. **MedGemma 1.5** — open-weight, already medical-tuned by Google,
   ~91% MedQA. The fast sidestep: drop in, no training needed.
3. **Fine-tune Nemotron-Nano on a user-built corpus.** Slower path,
   but you own the corpus, you own the weights, you own the
   provenance trail.

User picked 3. Reason (inferred from prior turns + memory): the
prism42 disclosure posture rewards owning the provenance trail end-
to-end, and a Brandon-Dent-MD-curated corpus is a defensible
medical-evidence source in a way that a Google-trained model is not.

## 2. Anthropic policy posture

This is **not** prohibited by Anthropic's Usage Policy
(https://www.anthropic.com/legal/aup). The policy restricts using
**Claude's outputs** to train, distill, or develop a competing model.
Nemotron-Nano is NVIDIA's open-weight model under their Open Model
License; fine-tuning it on a non-Claude corpus is not the prohibited
shape. The lines that matter:

1. **Don't seed the corpus from Claude** — no synthetic Q&A
   generated by calling the Anthropic API, then trained into Nemotron
   to "distill" Claude's medical reasoning. That's the bright line.
2. **Don't market the result as Claude-class.** Position the
   fine-tune as a Nemotron medical fine-tune.
3. **Architectural separation already correct in prism42.** CLAUDE.md
   §1 frames Claude Opus 4.7 as the **auditor / dialectic harness**;
   Nemotron is the **production runtime**. Two roles, no competition.

If certainty is needed for a specific corpus or training plan,
escalate via the Anthropic feedback channel already in use under the
prism42 disclosure posture.

## 3. Corpus sourcing checklist (no Claude outputs)

Acceptable sources for the medical corpus:

- [ ] Peer-reviewed medical literature (PubMed, journal full-text
      via institutional access).
- [ ] AHA / NHTSA / NHS-published clinical guidelines + protocols
      (already cited in GEDP v0.1).
- [ ] StatPearls / NBK reference articles (NCBI Bookshelf).
- [ ] FHIR R4 resource bundles (synthetic via OpenEM corpus, or
      MIMIC-IV / MIMIC-III with DUA compliance).
- [ ] Brandon-Dent-MD-authored / annotated clinical case material.
- [ ] Public PSAP transcripts and 911 dispatcher protocols (from
      the prism42 corpus already curated).
- [ ] HealthBench Hard / MedQA / PubMedQA / MedAgentBench evaluation
      corpora — for **eval, not training** (training on eval data is
      contamination).

Forbidden sources:

- [x] **Anthropic API outputs** — no Claude-generated synthetic
      Q&A in the training set.
- [x] Anything where the license forbids derivative-model training.
- [x] PHI without IRB / DUA / consent.
- [x] IAED-licensed protocol content (the prism42 stack already
      excludes this; preserve the boundary).

## 4. Training scope (assistant does NOT execute)

This subsection is intentionally a sketch, not an instruction. The
user owns every step.

- **Base model:** `nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-NVFP4` (or
  the BF16 base before NVFP4 quantization, depending on which
  fine-tune target the user picks).
- **Method:** LoRA / QLoRA likely sufficient for a domain adapter at
  this scale; full fine-tune is not required for clinical-text
  domain shift unless quality is insufficient.
- **Hardware:** B300 (`chemical-black-lungfish` Helsinki/Verda pod
  is sized for this) for short experiments; longer runs likely need
  a multi-GPU rental.
- **Eval gates:** baseline-vs-fine-tune paired-design comparison on
  HealthBench Hard, MedQA, PubMedQA, MMLU-Medical-6 + MedAgentBench
  per the existing prism42 benchmark grammar (CLAUDE.md §4). Same
  CI-excludes-zero gate.

**Until the corpus exists, no training.** This brief sits idle until
the user has a curated v0.1 corpus. The training session itself is
not on the assistant's path.

## 5. What this brief is NOT

- Not a green-light to train. It's a planning sketch.
- Not a replacement for the standalone TRT-LLM brief — serving and
  fine-tuning are independent decisions. Even an unfine-tuned
  Nemotron-Nano-30B-A3B-NVFP4 still benefits from the TRT-LLM
  serving runtime swap.
- Not a promise of "medical accuracy gains." The quality of any
  fine-tune is bounded by the corpus; bad data = worse model.
